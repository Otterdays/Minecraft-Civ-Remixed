# Changelog

All notable changes to this project are documented here.

## [Unreleased]

### Fixed
- **`/otter` small-window usability:** when the Minecraft window / GUI viewport is too cramped for the full 480×268 civ panel, the client now shows a compact fallback prompt instead of rendering a clipped, barely usable full menu. The fallback explains how to recover (enlarge the game window or lower GUI Scale) and still exposes quick buttons for `/money`, `/job`, and `/job list`.
- **Jobs HUD discoverability / stale docs mismatch:** `/otter` → Jobs now shows a preview of the actual HUD bar plus one-click join buttons for miner / lumberjack / farmer / fighter, so the menu no longer looks like "HUD controls with no bar." The live overlay now attaches after `VanillaHudElements.INFO_BAR` instead of the experience-level text layer, and docs were corrected to the real config path `config/project_ooga/jobs_hud.properties` (the older `config/fpsmod/jobs_hud.properties` note was stale after the mod-id rename).
- **Jobs HUD long-label overrun / scaling edge case:** the live bar no longer uses a naive fixed text budget. It now measures the left label and right XP text, expands width when the screen allows it, truncates safely only when space is genuinely tight, reuses the same sizing path in the `/otter` preview, and clamps the final bar position back on-screen after offsets are applied. This fixes the `LUMBERJACK` overlap seen in the HUD screenshots and makes `scale` behave like a layout footprint control instead of a "shrink until text collides" trap.
- **Reward chat wording / jobs feedback:** passive payouts no longer print hardcoded `"(mining)"` or `"(combat)"` labels. Economy messages now say `+N coins`, which avoids falsely implying the player earned miner XP from unrelated rewarded blocks like logs. Matching active jobs emit a separate XP/progress line (for example `[miner] +5 xp · Lvl 0 · 5/100`), and level-up lines still fire only on threshold crossings.

### Changed
- **Broad entity reward prefill / hostile-only fighter split:** the default bundled reward tag `otters_civ_revived:currency_mobs` now seeds `entity_values.json` from a much wider living-entity roster instead of only the old hostile list, giving operators a fuller editable per-entity payout sheet out of the box. To avoid widening fighter progression by accident, a new bundled companion tag `otters_civ_revived:hostile_mobs` now holds the hostile-only set and `otters_civ_revived:job/fighter_mobs` points there instead of mirroring the broader economy reward tag. Per user request, `minecraft:sulfur_cube` is included in both bundled entity tags as an optional `required: false` entry so future runtimes can resolve it without breaking current ones.
- **Agent workflow / Cursor subagent cost control:** added always-on rule `.cursor/rules/subagent-cost-control.mdc` and matching `AGENTS.md` / `CLAUDE.md` guidance so agents do not request explicit paid or Opus subagent models unless the user explicitly asks. Project guidance now prefers inherited/`Auto` behavior, future custom subagents should use `model: inherit`, and the docs note that built-in Cursor subagents are still steer-only so the safest no-credit path is keeping the main chat on `Auto` with Max Mode off.
- **Entrypoint classes renamed from `FpsMod`/`FpsModClient` to `OogaMod`/`ProjectOogaClient`** — resolves class name collision with the standalone FPS overlay mod. Fabric Loader's Knot classloader uses parent-first delegation, so two jars defining `com.fpsmod.FpsMod` causes the wrong copy to load at runtime (the FPS mod's version wins, our init code never runs). Renaming to unique names eliminates the collision entirely. `fabric.mod.json` entrypoints updated accordingly. All internal code references `OogaMod.MOD_ID`/`OogaMod.LOGGER` etc. (54 references across 47 files). Test class renamed to `OogaModTest`.

- **Mod ID changed from `fpsmod` to `project_ooga`** — resolves mod ID conflict with the standalone FPS overlay mod (the original template this project was forked from). Both mods can now coexist in the same modpack. The legacy FPS HUD overlay (`FpsHudOverlay`, `FpsHudScreenButton`, `FpsHudConfig`) is deprecated, disabled, and no longer registered as a client entrypoint (`ProjectOogaClient` replaces `FpsModClient`). The standalone FPS overlay mod handles FPS display. Networking and saved-data namespaces updated: `project_ooga:job_status`, `project_ooga:join_attendance`. Icon moved from `assets/fpsmod/` to `assets/project_ooga/`. Package names remain `com.fpsmod.*` (internal only, no mod ID conflict).

### Added
- **Atomic write crash safety for wallet + jobs + reward-config persistence:** new `AtomicFileWriter` (`com.fpsmod.io`) writes to a `.tmp` sibling file, fsyncs via `FileChannel.force(true)`, then atomically renames via `Files.move(ATOMIC_MOVE, REPLACE_EXISTING)`. If the JVM crashes mid-write only the `.tmp` is lost; the target file is untouched. `FileWalletStore.save()` and `FileJobsStore.save()` use `writeAtomicallyWithBackup()` (preserves a `.bak` as last-known-good insurance — restored if the atomic rename itself fails). `RewardRulesLoader.persistSortedLongMapJson()` and `writeDefaults()` use `writeAtomically()`. Legacy wallet migration (`migrateLegacyWalletIfNeeded`) now reads → atomic-writes → deletes legacy, safe from partial migration. Stale `.tmp` files are cleaned and logged on `load()`. `WalletService.persist()` and `JobsService.persist()` are now `synchronized` to prevent concurrent `.tmp` races. Backed by `AtomicFileWriterTest` (11 tests: normal write, crash simulation, stale cleanup, concurrent writes, backup rotation, large content) and `FileWalletStoreAtomicTest` (6 tests: round-trip, stale cleanup, overwrite, null-safe hints). See `DOCS/ARCHITECTURE.md` amendment.

### Fixed
- **THE actual root cause — three stacked bugs (2026-05-11):** the runtime warning `registry holds 374 bound tags total` proved the registry was fully populated yet our specific tag ids returned zero. Investigation against the installed `minecraft-common-deobf-26.1.2.jar`:
  - **Resource paths were plural** (`data/otters_civ_revived/tags/blocks/`, `tags/entity_types/`) — Minecraft 1.21+ loads tags only from **singular** directories (`tags/block/`, `tags/entity_type/`). The mod's bundled `currency_blocks` / `currency_mobs` tags were silently dropped by `TagLoader`. Resource files moved to the singular paths so `TagLoader` picks them up.
  - **`minecraft:hostile` does not exist as an entity-type tag in vanilla 1.21+** (verified by enumerating `data/minecraft/tags/entity_type/` in the deobf jar; no such file). Default `RewardRules.entityTag` changed from `"minecraft:hostile"` → `"otters_civ_revived:currency_mobs"` (the mod's own bundled tag).
  - **`currency_mobs.json` previously referenced `#minecraft:hostile`** (recursive include of the same non-existent tag) so even after fixing the path it would have stayed empty. Replaced with an explicit list of every vanilla hostile mob (zombie/skeleton/creeper/spider/enderman/witch/slime/magma_cube/phantom/blaze/ghast/piglin family/wither/warden/illager family/guardian family/shulker/silverfish/endermite/breeze).
- **Tag expansion now uses `Registry.getTagOrEmpty(TagKey)` on `BuiltInRegistries`** instead of the level's `registryAccess()` `HolderLookup` view. Diagnosed from runtime logs: `level.registryAccess().lookupOrThrow(Registries.X).get(TagKey)` returned `Optional.empty()` for every tag — even vanilla `minecraft:hostile` — meaning the `RegistryAccess.Frozen` view doesn't carry tag bindings in this Minecraft version. `TagLoader` binds datapack tags onto the **static** `BuiltInRegistries` registry's tag map during resource reload, so the static `getTagOrEmpty` path is the reliable enumeration. Also registers an `END_DATA_PACK_RELOAD` handler in `FpsMod` so the prefill re-runs after `/reload`. Diagnostic now also logs the total bound-tag count when a lookup yields zero entries.
- **Tag expansion rewritten to use direct `HolderSet.Named` lookup** instead of iterating every registry entry and asking `holder.is(TagKey)`. Both block and entity prefill now call `level.registryAccess().lookupOrThrow(Registries.X).get(tagKey)` and iterate the returned `HolderSet.Named` members, then map each holder's value back to its identifier via `BuiltInRegistries.X::getKey`. This is the canonical way to enumerate tag members in modern Minecraft and reliably reflects datapack-bound tags at `SERVER_STARTED`. Adds explicit `INFO` log on success (entry count) and `WARN` on empty/missing tag — operators can confirm in `logs/latest.log` whether the prefill found members.
- **`block_values.json` / `entity_values.json` were persisting as empty `{}`** when the default `otters_civ_revived:currency_blocks` tag (and any other datapack-defined block tag) was used. Block tag expansion previously walked `BuiltInRegistries.BLOCK` and called `BlockState.is(TagKey)` — that path checks the static built-in holders, but in modern Minecraft datapack tag bindings live on the **server's live registry** layer. `RewardTagExpansion.payoutsForTaggedBlocks` now takes a `ServerLevel` and resolves membership via `level.registryAccess().lookupOrThrow(Registries.BLOCK)`, mirroring the entity path. Also: tag expansion no longer bails when the flat `blockReward`/`entityReward` is `0` — empty/zero payouts still get a prefilled id list so operators have something concrete to edit. Logs a warning when zero blocks/entities resolve for a tag id. End user-visible: editing `block_values.json` / `entity_values.json` now actually has content to customize.
- **`RewardRulesLoader.writeDefaults`**: append a trailing newline when generating the default `config/otters_civ_revived/rewards.json`, matching the sibling `block_values.json` / `entity_values.json` writer (`persistSortedLongMapJson`). Cosmetic POSIX-friendly fix; no behavior change.
- **Test coverage:** added direct unit tests for `RewardRulesLoader.composeEffectiveIdMap` (production merge path used by `finalizeRewardsForRunningServer`) covering tier1 → inline → sibling precedence, sibling-empty persistence with trailing newline, and the "do not overwrite existing sibling file" invariant. Closes the gap where only the test-only `mergeExternalValueFiles` helper was exercised.

- **`DOCS/modrinth-description.md`** restored at canonical path (`STYLE_GUIDE`, `SUMMARY`, `index.html` reference); preservation header added. Parallel copy under `DOCS/Ryan-Made-Docs/` may remain for author drafts.
