Otters Civ. Revived

Unofficial reference page for the Fabric mod (technical id: fpsmod). Repository codename: Project OOGA / Minecraft Civ Remixed.

Otters Civ. Revived is a server-oriented Fabric mod that introduces a persistent wallet, chat commands for balances, configurable payouts for breaking certain blocks and killing tagged mobs, and a bundled legacy FPS overlay from an earlier template. Long-term design targets include guild-based territory, social systems (friends, private messages), deeper economy (transfers, ledger), and jobs with leveled traits—see Target feature set.

Features (current build)

Server

  • /otter — in-game command list + pointer to rewards config.
  • /money and /money set <player> <amount> — read/set wallet (set is bootstrap; permissions TBD).
  • Persistent balances in config/fpsmod/wallet.properties.
  • Mining payouts for blocks in otters_civ_revived:currency_blocks (see datapack + rewards.json).
  • Combat payouts for mobs matching the configured entity tag (default #minecraft:hostile); direct player melee attribution only in this version (no bow/trident credit).
  • Tunable config/otters_civ_revived/rewards.json (amounts, cooldowns, creative/spectator skip, dimensions, announcements).
  • JobsHooks seam present; no job systems wired yet.

Client

  • Optional legacy FPS HUD (readout + GUI toggle); config/fpsmod/hud.properties.

Not in this release

  • Only wallet + command bootstrap + passive payouts are live—see Target feature set for the full design (transfers, guilds, social, jobs).
  • Missing today: /pay, permissions on /money set, ledger, friends, PM, guild claims/protections, job trees, shops, SQLite—see DOCS/ROADMAP.md.

Target feature set (design goals)

Below is the intended scope of Otters Civ. Revived: a civ-style multiplayer layer on Fabric. Ship order and detail follow the repo roadmap; this is not a commitment timeline. Group territory is framed as guilds (faction-like mechanics without using “factions” as the product label).

Economy

  • Money — server-authoritative currency tied to rewards, trades, and sinks.
  • Wallets — persistent per-player balances (prototype: properties file; later storage may migrate).
  • Transfers — paying other players (/pay-style commands), limits, optional fees, and immutable ledger entries (planned).

Social

  • Friends lists — player-curated lists for social features (invites, trust, or soft permissions—exact UX TBD).
  • Private messaging — direct messages between players separate from public chat (design target).

Guilds & territory

  • Guilds — organized groups with membership, roles, and a treasury hook consistent with the economy layer.
  • Chunk claims — assign chunks (or regions) to a guild; overlap and raid rules configurable.
  • Protections — enforce build/break/container use inside claimed space; outsiders handled via diplomacy/war-style rules (design target).

Jobs, professions & traits

  • Jobs / professions — player-picked roles that contextualize how you earn (mining, farming, crafting, etc.).
  • Skill-based rewards — income and bonuses tied to job-related actions, not only global block/mob tags.
  • Traits & levels — per-job progression (levels, perks, or trait lines)—integrates with the mod’s planned JobsHooks pipeline for stacking modifiers on rewards.
Current releases implement only wallets, /money, tag-based mining/combat grants, and a no-op jobs hook. Guilds, messaging, friends, /pay, and profession trees are not yet in the JAR.

The mod declares environment * (client + server): install on the server for economy and payouts; clients only need the JAR if you want the optional FPS HUD overlay.

Commands

Commands register on the logical server (including single-player integrated servers). The in-game analogue of this article is /otter, which prints a short synopsis.

/otter
Lists the mod’s slash commands and points operators to the rewards configuration path. No arguments.
/money
Requires an executing player context. Sends the requesting player’s wallet balance in chat.
/money set <player> <amount>
Sets the target player’s balance to amount (≥ 0). Intended for admin bootstrap and balancing; broader permission gates are planned.

Economy & persistence

Balances are keyed by player UUID and held in memory at runtime with writes to config/fpsmod/wallet.properties. Administrative /money set performs a full replace of the stored balance for that UUID; gameplay rewards call additive credit paths.

The wallet file is a Java .properties map: each key is a player UUID string, each value is a non‑negative decimal integer balance. The store comments the file as Project OOGA wallet balances on write. Invalid lines are skipped with a logged warning.

The rewards file (rewards.json) configures tags and payout sizes; wallet storage remains under config/fpsmod/.

Passive rewards

Mining and combat payouts are controlled by JSON at config/otters_civ_revived/rewards.json (created with defaults when missing). Restart the client or server after editing; the file is read once at startup.

What counts: tags and datapacks

Which blocks and mobs pay is not hard-coded in that JSON as a block list. The config points at tag identifiers; the tag contents come from Minecraft’s tag system (vanilla + mod + user datapacks).

  • blockTag — block tag id (e.g. otters_civ_revived:currency_blocks). A broken block pays only if it matches this tag. Point it at any valid block tag, including vanilla (e.g. minecraft:iron_ores).
  • entityTag — entity type tag id (default minecraft:hostile). A kill pays only if the mob’s type is in that tag. You may use the mod’s otters_civ_revived:currency_mobs or a custom tag from your own datapack.
  • Membership — The mod ships tag JSON under data/otters_civ_revived/tags/… inside the JAR. Server owners can add world/server datapacks to merge extra entries into those tags or define new tags, then set blockTag / entityTag to those ids in rewards.json.
  • Amounts — blockReward and entityReward are single flat values for all matches in the tag (no per-block or per-mob table in this version).

Other keys in the same file: enabled, per-action cooldowns, skip creative/spectator, dimensionBlacklist (dimension ids), announceRewards. Defaults shipped in code use block tag otters_civ_revived:currency_blocks and hostile mobs via minecraft:hostile.

Kill credit in this build favors direct player hits; bows, tridents, and other indirect kills are not credited yet.

Bundled datapack tags (default JAR)

These are the shipped entries merged into the mod’s namespaces. Operators can supersede or extend them with world/server datapacks; rewards.json decides which tag ids participate in payouts.

otters_civ_revived:currency_blocks

Block ids and nested tag references as in the repo JAR (data/otters_civ_revived/tags/blocks/currency_blocks.json). Ore lines expand every block in each vanilla ore tag.

  • minecraft:stone, minecraft:granite, minecraft:diorite, minecraft:andesite
  • minecraft:deepslate, minecraft:tuff, minecraft:calcite, minecraft:dripstone_block
  • minecraft:netherrack, minecraft:blackstone, minecraft:basalt, minecraft:end_stone
  • #minecraft:coal_ores, #minecraft:iron_ores, #minecraft:copper_ores, #minecraft:gold_ores
  • #minecraft:redstone_ores, #minecraft:lapis_ores, #minecraft:diamond_ores, #minecraft:emerald_ores

otters_civ_revived:currency_mobs

The shipped entity-type tag currently includes only #minecraft:hostile. The default config’s entityTag is still the literal id minecraft:hostile; you may point entityTag at otters_civ_revived:currency_mobs in rewards.json if you want a single customization surface that matches the datapack folder name.

config/fpsmod/wallet.properties

UUID keys and long balances; created automatically when balances exist. Prefer using /money set for edits rather than hand-editing unless you maintain backups.

config/fpsmod/hud.properties

Client-only: showFpsHud=true|false for the legacy FPS overlay. Read failures default to visible HUD for the session.

Client features (detail)

A legacy FPS counter and pause/inventory overlay toggle persists from the repository’s older template (FpsHudOverlay / HUD config under config/fpsmod/). It is optional ornamentation for the owning client install and unrelated to civ economy semantics.

Operational limits

  • One rewards config directory per launcher instance applies to integrated server usage; wallet file scope follows the same Fabric config root.
  • Tag-based payouts do not imply “worldgen-only” blocks; duplication farming is deferred to roadmap hardening.
  • Future professions may subscribe via the JobsHooks extension seam (currently no-op).
