Otters Civ. Revived (codename: Project OOGA) is a Fabric mod that keeps a saved wallet per world, adds slash commands you type in normal Minecraft chat, shows join system messages (full onboarding the first time you connect to that save, a shorter welcome back ~ line when you’ve been there before), pays you for mining or fighting mobs (see rewards.json), and ships a jobs HUD bar above vanilla XP when you have an active job. Long-term design: guild-style land, social tools, transfers, ledger, deeper jobs—see Target feature set.

Features (current build)

Shipped today: money on disk, chat commands, join tips, configurable mining/kill payouts, and a jobs HUD bar with in-game controls.

World side (money & rules)

  • Friendly lines in chat when you join: first visit on this world save = three system lines (branding, /otter, /money, rewards pointer). Returning players see a styled welcome back ~ display name plus a shorter refresher (persisted per save, not in config/).
  • /otter — opens a stylized in-game menu (sidebar tabs: Home · Wallet · Jobs · Rewards · Civ · Help) with quick-action buttons for /money, jobs/HUD controls, and one-click links that open the config files (rewards.json, block_values.json, entity_values.json, wallet.properties) in your OS file manager. On vanilla clients without the mod, the server-side fallback prints the same info as chat lines.
  • /money checks your balance (everyone). /money set <player> <amount> changes someone’s total and requires vanilla gamemaster-style command permission (same band as cheat commands / OP)—see permissions.
  • Wallets save under config/otters_civ_revived/wallet.properties (same folder as rewards; legacy config/fpsmod/wallet.properties is moved here on first load).
  • Mining: pay when you break blocks in blockTag, unless that block has its own line in blockRewards (defaults ship with currency_blocks).
  • Combat: pay on kills that match entityTag (default: hostile mobs), with optional per-mob amounts in entityRewards; this build only credits direct melee—not bows or tridents yet.
  • Tune everything in config/otters_civ_revived/rewards.json (cooldowns, gamemode skips, dimensions, chat announcements, tag ids, per-id maps).
  • /job, /job list, /job join <slug>, and /job leave drive the first jobs slice (miner, lumberjack, farmer, fighter). The jobs HUD only appears while you have an active job selected.

Client

  • Jobs HUD overlay — compact bar above vanilla XP showing icon, job slug, level, and XP progress. Tune it from /otter → Jobs (preview, show/hide, X/Y nudges, scale, reset, one-click join buttons). Client config lives at config/project_ooga/jobs_hud.properties.
  • (deprecated) Legacy FPS HUD overlay — disabled in project_ooga. The standalone FPS overlay mod (original template) handles FPS display.

Not in this release

  • Only wallet + command bootstrap + passive payouts + jobs MVP are live—see Target feature set for the broader civ stack (transfers, guilds, social, deep professions).
  • Missing today: /pay, granular mod/plugin permission layers (beyond vanilla gamemaster for /money set), ledger, friends, PM, guild claims/protections, deeper job trees/perks, shops, SQLite—see DOCS/ROADMAP.md.

Target feature set (design goals)

Plain picture: one economy everyone trusts, guilds that own land together, jobs that change how you earn—not just the passive mining/kill loop you have now.

Below is the intended scope of Otters Civ. Revived: a civ-style multiplayer layer on Fabric. Ship order and detail follow the repo roadmap; this is not a commitment timeline. Group territory is framed as guilds (faction-like mechanics without using “factions” as the product label).

Economy

  • Money — one authoritative balance the world enforces (not something clients can fake), tied to rewards, trades, and sinks.
  • Wallets — persistent per-player balances (prototype: properties file; later storage may migrate).
  • Transfers — paying other players (/pay-style commands), limits, optional fees, and immutable ledger entries (planned).

Social

  • Friends lists — player-curated lists for social features (invites, trust, or soft permissions—exact UX TBD).
  • Private messaging — direct messages between players separate from public chat (design target).

Guilds & territory

  • Guilds — organized groups with membership, roles, and a treasury hook consistent with the economy layer.
  • Chunk claims — assign chunks (or regions) to a guild; overlap and raid rules configurable.
  • Protections — enforce build/break/container use inside claimed space; outsiders handled via diplomacy/war-style rules (design target).

Jobs, professions & traits

  • Jobs / professions — player-picked roles that contextualize how you earn (mining, farming, crafting, etc.).
  • Skill-based rewards — income and bonuses tied to job-related actions, not only global block/mob tags.
  • Traits & levels — per-job progression (levels, perks, or trait lines)—integrates with the mod’s planned JobsHooks pipeline for stacking modifiers on rewards.
Current releases implement only wallets, /money, join tips, tag-based mining/combat grants with optional per-id payout maps, and a no-op jobs hook. Guilds, messaging, friends, /pay, and profession trees are not yet in the JAR.

Install note: Put the mod on the machine that hosts the world (single-player = your own game; multiplayer = the host’s mods folder). Players only need the client JAR if they want the jobs HUD overlay or /otter GUI extras—money and commands still run from the world host.

Slash commands (in-game chat)

Simple explanation: Press your chat key (often T), type a line that starts with / (for example /money), press Enter. That works while you are playing inside a world—single-player or multiplayer. You are not using a separate “server program” window; “server” in modding just means the part of Minecraft that remembers money and runs the command.

Try /otter first — on a client with this mod installed it opens a stylized in-game menu (tabbed: Home · Wallet · Jobs · Rewards · Civ · Help) with one-click buttons to run /money, preview the jobs HUD, join a job, or open config files. On vanilla clients it prints a short chat help list instead.

Players also see automatic system messages when they connect (not a slash command). The first time your account is seen on this world save, you get three lines: Otters Civ. Revived branding, /otter, /money, and a pointer to passive rewards / rewards.json. Later joins get a gold/aqua welcome back ~name line and a condensed command reminder—stored per save as project_ooga:join_attendance (overworld saved data), separate from wallet.properties.

/otter
Shows which commands exist and reminds operators where rewards.json lives. No arguments.
/money
You must be a player in the world (same as vanilla chat commands). Prints your balance.
/money set <player> <amount>
Replace that player’s saved balance with amount (≥ 0). Requires Minecraft gamemaster-level command permission (OP-equivalent for cheat-style commands)—see below. Future builds may expose explicit mod nodes / LuckPerms-style checks.
/job, /job list, /job join <slug>, /job leave
Pick one active job at a time (miner, lumberjack, farmer, fighter). The jobs HUD bar uses that active selection and only renders while a job is active.

Who can run which command (today)

Open to players: /otter (help only), /money with no arguments (read your balance in chat). Passive mining/kill payouts have no slash command—they follow rewards.json plus optional block_values.json / entity_values.json (merged at startup).

Gamemaster / OP band only: /money set <player> <amount> checks CommandSourceStack.permissions() for Permission.HasCommandLevel(GAMEMASTERS)—same tier vanilla uses for many built-in cheats. Dedicated console and suitably permissioned integrations still work as the server configures them.

Future: a dedicated Otters Civ. permission apparatus (explicit string ids, optional Fabric Permissions API / server plugins, non-OP econ moderators). Sketched under Permissions apparatus (planned) in DOCS/ROADMAP.md and M1’s admin-permission backlog.

Economy & persistence

Think “bank file on disk”: one number per player UUID, updated when you earn or an admin sets your balance.

Balances are keyed by player UUID and held in memory at runtime with writes to config/otters_civ_revived/wallet.properties. Administrative /money set performs a full replace of the stored balance for that UUID; gameplay rewards call additive credit paths.

On disk the wallet is UTF‑8 text: optional # Name: … hint lines precede each uuid=balance line (same semantics as Java .properties for the key=value part). Loader accepts older files without name lines. Comments on write explain the format; malformed balance lines log a warning and are skipped when possible.

rewards.json, optional block_values.json / entity_values.json, and wallet.properties live under config/otters_civ_revived/. The grandfathered FPS overlay is deprecated (handled by the standalone FPS overlay mod).

Passive rewards

Breaking the right blocks and killing the right mobs silently adds coins. What “right” means is editable JSON—often via tags Datapacks already understand—with optional exceptions per exact block/mob ID.

Mining and combat payouts load from JSON at config/otters_civ_revived/rewards.json, then merge sibling block_values.json and entity_values.json whole-file maps (see below). All are created with defaults when missing. Restart the client or server after editing; the files are read once at startup.

What counts: tags and datapacks

Which blocks and mobs pay is not hard-coded in that JSON as a block list. The config points at tag identifiers; the tag contents come from Minecraft’s tag system (vanilla + mod + user datapacks).

  • blockTag — block tag id (e.g. otters_civ_revived:currency_blocks). Used for the fallback blockReward when that block’s registry id is not listed in blockRewards. Point it at any valid block tag.
  • entityTag — entity type tag id (default minecraft:hostile). Used for entityReward fallback when the mob’s id is missing from entityRewards. Your own datapack tags are fine.
  • Membership — The mod ships tag JSON under data/otters_civ_revived/tags/… inside the JAR. Server owners can add world/server datapacks to merge extra entries into those tags or define new tags, then set blockTag / entityTag to those ids in rewards.json.
  • Per-block / per-mob payouts — Objects blockRewards and entityRewards in rewards.json map registry ids (e.g. "minecraft:diamond_ore": 40, "minecraft:zombie": 2) to whole-number payouts. Keys are validated and normalized at load; invalid ids are skipped with a server log warning.
  • Dedicated value files — block_values.json / entity_values.json beside rewards.json normally contain one line per rewarded block/hostile mob; Otters Civ. seeds them automatically on SERVER_STARTED whenever a sibling file parses to zero keys ({}, invalid JSON counts as zero). Rows default to whichever blockTag / entityTag resolves to plus flat blockReward / entityReward. Inline blockRewards / entityRewards overrides win on shared keys.
  • Fallback amounts — Once maps are merged, if the broken block’s id is not listed, the payout is blockReward only when that block matches blockTag. Same pattern for kills: merged entityRewards wins, otherwise entityReward when the victim’s type matches entityTag. Listed ids pay their configured amount even without tag membership.

Other keys in rewards.json: enabled, optional blockRewards / entityRewards, per-action cooldowns, skip creative/spectator, dimensionBlacklist (dimension ids), announceRewards. Defaults shipped in code target otters_civ_revived:currency_blocks and #minecraft:hostile; sibling value files populate from those tags automatically the first time the server starts with empty maps.

Kill credit in this build favors direct player hits; bows, tridents, and other indirect kills are not credited yet.

config/otters_civ_revived/block_values.json

Whole-file map: block registry id (namespace:path) → non‑negative payout. On Fabric’s logical-server started callback, Otters Civ. expands whatever blockTag points at (including nested datapack references like #minecraft:diamond_ores), assigns each matched block flat blockReward, unions inline blockRewards from rewards.json, and—only while this file parses to zero keys—persists sorted UTF‑8 JSON you can skim like a spreadsheet.

If this file already has keys, startup only merges—it never wipes your edits. Broken JSON logs an error and is treated empty (then regenerated if still “no sibling keys”).

config/otters_civ_revived/entity_values.json

Same pattern for entity type ids (#minecraft:hostile resolves to dozens of vanilla mobs) using flat entityReward defaults when prefilling.

Again: regenerated only while the sibling file has no surviving keys—existing operator files remain authoritative overlays.

If the sibling files persist as { }: the mod prefills on both SERVER_STARTED and END_DATA_PACK_RELOAD, so running /reload (or deleting the empty files and restarting) re-fills them. logs/latest.log will print [otters_civ_revived] block tag … resolved N entries on success, or a warning with the registry's total bound-tag count if the tag id can't be resolved (typo, missing datapack, etc.).

Adding your own blocks & mobs

Three ways, pick the one that fits how permanent the change should be. All three coexist; later layers override earlier ones for the same id.

  1. Edit block_values.json / entity_values.json directly — easiest. Add lines like "minecraft:ancient_debris": 500, or "yourmod:custom_ore": 75, anywhere in the JSON object, save, and /reload (or restart). Per-id payouts always win over tag fallback, so this works for blocks/mobs that aren't in any tag at all. Keep keys lowercase namespace:path; invalid ids log a warning and are skipped.
  2. Inline in rewards.json — same idea, different file. Add entries under blockRewards or entityRewards: 
    {
  "blockRewards": {
    "minecraft:ancient_debris": 500,
    "yourmod:custom_ore": 75
  },
  "entityRewards": {
    "minecraft:warden": 250
  }
}
    At startup the sibling value files merge on top, so if the same id appears in both files, the sibling file wins. Use whichever feels natural — rewards.json for "config I check in," sibling files for "ops dial-tuning."
  3. Extend the bundled tag via a server datapack — for entries that should automatically appear in regenerated sibling files and inherit the flat blockReward / entityReward:
    1. Drop a pack.mcmeta + data/otters_civ_revived/tags/block/currency_blocks.json into <world>/datapacks/<your_pack>/ (note the singular block / entity_type directory names — plural is silently ignored by MC 1.21+).
    2. Contents: { "replace": false, "values": ["yourmod:custom_ore", "#yourmod:custom_ore_tag"] }. Nested #tag references resolve recursively.
    3. /reload — the prefill re-runs on END_DATA_PACK_RELOAD; if the sibling file has no keys yet, your entries appear in the regenerated JSON at blockReward amount.
    Alternatively, change blockTag / entityTag in rewards.json to point at a different tag id entirely (vanilla, modded, or one you defined).

Precedence (high → low) when a block breaks or mob dies: sibling value file entry → rewards.json inline map entry → tag membership with flat reward → no payout. So any explicit per-id line in either JSON beats tag-wide fallback.

config/otters_civ_revived/wallet.properties

Who is who: Each entry is still uuid=balance (that pair is authoritative). When the mod knows your in-game display name, it writes the line above as # Name: SomePlayer so operators can skim the file—safe to edit comments; only the uuid=balance line sets money. Names refresh on join, /money, /money set, mining, and combat payouts.

Created automatically when balances exist. Older builds kept this under config/fpsmod/; that file moves here once and logs [otters_civ_revived] Migrated…. Prefer /money set for balance edits unless you keep backups.

New (2026-05-12): Crash safety: all config writes (wallet.properties, jobs.properties, rewards.json, block_values.json, entity_values.json) now use an atomic write pattern — content goes to a .tmp sibling file, fsyncs, then atomically renames to the target. If the game crashes mid-write, the previous file is untouched. A stale .tmp leftover is cleaned and logged on the next load(). Previous-good copies are preserved as .bak files as additional insurance. See DOCS/ARCHITECTURE.md amendment for full detail.

config/project_ooga/jobs_hud.properties

Client-only jobs HUD config: visible=true|false, offsetX, offsetY, and scale. The file is written when you use the /otter → Jobs controls. Default behavior: visible, centered, 12 px above the vanilla XP bar, scale 1.0.

Deprecated: the old FPS-only config remains at config/fpsmod/hud.properties for the standalone FPS overlay lineage, but Project OOGA no longer registers that overlay.

Client features (detail)

Jobs HUD: Project OOGA draws a compact jobs bar above vanilla XP whenever the local player has an active job and the HUD is enabled. The /otter → Jobs tab now includes a preview of that bar, direct join buttons for all four jobs, and the usual visibility / offset / scale controls so you can see what the overlay is supposed to look like before closing the menu.

Scaling behavior: the bar width is now content-aware. Longer labels such as LUMBERJACK and larger XP strings no longer overrun the right-hand text at default settings; the HUD expands when it has room, truncates safely only when it must, and clamps back on-screen after offsets are applied.

Deprecated: The legacy FPS counter and overlay toggle from the repository's original template (FpsHudOverlay / HUD config under config/fpsmod/). The standalone FPS overlay mod handles FPS display; Project OOGA's client UX is the jobs HUD + civ menu.

Operational limits

  • One rewards config directory per launcher instance applies to integrated server usage; wallet file scope follows the same Fabric config root.
  • Join onboarding vs welcome-back is tracked per world save (overworld saved data project_ooga:join_attendance), not in global config/—a new single-player world or new server map always gets the full first-join messages again for each player UUID.
  • /money set is intentionally narrow: vanilla gamemaster permission only until a plugin-friendly permission layer ships (see command permissions).
  • Tag-based payouts do not imply “worldgen-only” blocks; duplication farming is deferred to roadmap hardening.
  • Future professions may subscribe via the JobsHooks extension seam (currently no-op).

Architecture

Project OOGA is structured as five domain layers built on top of the Fabric baseline. The current JAR only activates the economy surface; the layers below are the planned target shape.

LayerResponsibility
Data layerPersistent records for player, wallet, faction, claim, and shop entities; schema versioning and migration runner.
Domain servicesWallet, progression, faction governance, claim checks, market transactions—each behind a service boundary.
Event layerFabric event listeners routed through validation before calling domain services.
Interface layerCommands and Screen Handler UX; primary player-facing surface.

Runtime flow

  1. Player action emits game event.
  2. Validation checks permissions and limits.
  3. Domain service computes result.
  4. Transaction and state changes persist atomically.
  5. Player receives feedback via UI, chat, or action bar.
Reliability targets: no main-thread blocking I/O on hot gameplay paths; deterministic transaction updates with rollback-safe boundaries; explicit error paths for all state-mutating operations.

Installation

Client install is optional (jobs HUD + /otter GUI extras). Server install enables economy commands, jobs progression, and passive payouts.

  1. Install Fabric Loader >= 0.19.2 for Minecraft 26.1.2. The Fabric installer handles this.
  2. Drop Fabric API (0.146.1+26.1.2 or compatible) into your mods/ folder.
  3. Drop the mod JAR (project-ooga-1.0.0.jar) into mods/.
  4. Start the server (or client). Otters Civ. creates config/otters_civ_revived/ (rewards.json, wallet.properties) and, on clients using the jobs HUD, config/project_ooga/jobs_hud.properties.
  5. Optionally edit config/otters_civ_revived/rewards.json to tune payout amounts, tags, and cooldowns, then restart.
Java 25+ is required. The host of the world (single-player PC or multiplayer host) needs this mod installed for wallets, jobs, and payouts. Other players only need the JAR on their PC if they want the jobs HUD and the richer /otter client UI.

Building from source

Requires JDK 25+ and Git. The repo uses the Gradle wrapper—no separate Gradle install needed.

  1. Clone: git clone https://github.com/Otterdays/Minecraft-Civ-Remixed.git
  2. Build: ./gradlew build (Linux/macOS) or gradlew.bat build (Windows)
  3. Output JAR: BUILT/libs/project-ooga-1.0.0.jar

The build version and Minecraft target are pinned in gradle.properties. Match Fabric API and Loader to the pins there for a reproducible build.
